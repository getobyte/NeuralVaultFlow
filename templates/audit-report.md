# Audit Report Template

Template for NVC key `{project}:audit-{date}` — structured code audit reports with prioritized findings.

**Purpose:** Consistent audit output format that categorizes findings by severity and provides actionable fixes.

---

## File Template

```markdown
# Audit Report: {project-name}

**Date:** {date}
**Stack:** {stack-description}
**Scope:** {scope-description}

## Summary

| Severity | Count |
|----------|-------|
| CRITICAL | {critical-count} |
| HIGH | {high-count} |
| MEDIUM | {medium-count} |
| LOW | {low-count} |
| **Total** | **{total-count}** |

---

## CRITICAL

### {finding-id}. {finding-title}

**Category:** [Dead Code | Error | Consistency | Security | Quality]
**Location:** `{file-path}:{line-number}`
**Description:** [What the problem is — specific, not vague]
**Impact:** [What happens if this is not fixed — concrete consequence]
**Fix:** [Exact change to make — specific enough to implement]

---

## HIGH

### {finding-id}. {finding-title}

**Category:** [category]
**Location:** `{file-path}:{line-number}`
**Description:** [description]
**Impact:** [impact]
**Fix:** [fix]

---

## MEDIUM

### {finding-id}. {finding-title}

**Category:** [category]
**Location:** `{file-path}:{line-number}`
**Description:** [description]
**Impact:** [impact]
**Fix:** [fix]

---

## LOW

### {finding-id}. {finding-title}

**Category:** [category]
**Location:** `{file-path}:{line-number}`
**Description:** [description]
**Fix:** [fix]

---

## Recommendations

1. [Top priority action with justification]
2. [Second priority action]
3. [Third priority action]

---

*Generated by /nvc:audit on {date}*
```

---

## Field Documentation

| Field | Type | Required | Purpose | Example |
|-------|------|----------|---------|---------|
| `{project-name}` | variable | Yes | Project identifier | `my-saas-app` |
| `{date}` | variable | Yes | Audit date | `2026-03-26` |
| `{stack-description}` | variable | Yes | Tech stack summary | `Next.js 14, TypeScript, PostgreSQL` |
| `{scope-description}` | variable | Yes | What was audited | `Full codebase` or `src/api/ only` |
| `{finding-id}` | variable | Yes | Sequential finding number | `1`, `2`, `3` |
| `{finding-title}` | variable | Yes | Short finding name | `Hardcoded API key in config` |
| `[category]` | prose | Yes | One of 5 audit categories | `Security`, `Dead Code` |
| `{file-path}:{line-number}` | variable | Yes | Exact code location | `src/config.ts:42` |
| `[description]` | prose | Yes | What the problem is | `AWS_SECRET_KEY is hardcoded as string literal` |
| `[impact]` | prose | Yes (CRITICAL/HIGH) | Consequence of not fixing | `Credential leak if repo becomes public` |
| `[fix]` | prose | Yes | How to fix it | `Move to .env, reference via process.env.AWS_SECRET_KEY` |

## Section Specifications

### Summary Table
**Purpose:** Quick severity overview for prioritization.
**Quality check:** Counts match actual findings listed below.

### Finding Entries
**Purpose:** One actionable finding per entry.
**Quality check:** Every CRITICAL and HIGH has an impact statement. Every finding has a specific fix, not "fix this" or "improve this".

### Recommendations
**Purpose:** Top 3 actions ordered by impact.
**Quality check:** Each recommendation references specific findings.
