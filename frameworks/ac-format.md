## Purpose

Standard format for acceptance criteria used in plans generated by `tasks/plan.md`. Uses BDD-style Given/When/Then to define testable, observable behavior. Acceptance criteria are defined BEFORE implementation steps — they are the contract that steps must satisfy.

## The Format

```
Given [precondition — the state of the system before the action]
When  [action — what the user or system does]
Then  [outcome — what must be true after the action]
```

Each AC gets a unique identifier: `AC-1`, `AC-2`, etc.

## Rules

1. **AC before steps** — Define all acceptance criteria before writing implementation steps. Steps reference ACs, not the other way around.
2. **Observable behavior** — AC describes what a user or external system can observe. Not internal implementation details.
3. **Testable** — If you cannot write a test (manual or automated) for an AC, it is too vague. Rewrite it.
4. **One behavior per AC** — Each AC tests one thing. If you use "and" in the Then clause, consider splitting.
5. **No implementation details** — AC says WHAT happens, not HOW it's built.

## Good Examples

```
AC-1:
  Given user is authenticated
  When  accessing GET /api/invoices?page=1
  Then  receives 200 with paginated list of max 20 invoices

AC-2:
  Given user submits login form with valid credentials
  When  server processes the request
  Then  user receives a JWT token with 24h expiry

AC-3:
  Given product page is loaded
  When  user clicks "Add to Cart"
  Then  cart count increments by 1 and toast notification appears

AC-4:
  Given database has 10,000 records
  When  running the export function
  Then  CSV file is generated in under 5 seconds
```

## Bad Examples and Why

| Bad AC | Problem | Fix |
|--------|---------|-----|
| "The function fetchInvoices returns data" | Describes implementation, not behavior | "User sees paginated invoice list" |
| "It works correctly" | Not testable — what does "correctly" mean? | Define specific inputs and expected outputs |
| "Performance is good" | No metric — untestable | "Response time under 200ms for 95th percentile" |
| "The code is clean" | Subjective, not an AC | Remove — code quality is a review concern, not an AC |
| "User can do everything" | Too broad, untestable | Split into specific user actions |

## How Steps Reference ACs

Each step in a plan must reference which AC(s) it satisfies:

```xml
<step id="3" status="pending">
  <description>Implement pagination on invoices endpoint</description>
  <files>[src/api/invoices.ts]</files>
  <action>Add limit/offset pagination with default page size 20</action>
  <verify>GET /api/invoices?page=1 returns max 20 items with pagination metadata</verify>
  <done>AC-1 satisfied</done>
</step>
```

A step can satisfy multiple ACs. An AC can be satisfied by multiple steps working together. But every AC must be referenced by at least one step.

## Completeness Check

Before a plan is saved:
- Every AC has a unique ID
- Every AC follows Given/When/Then format
- Every AC is referenced by at least one step
- Every step references at least one AC
- No AC is vague or untestable
